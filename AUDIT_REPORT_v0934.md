# รายงานตรวจระบบ Evolution of Us v0.9.34

## สรุป

รุ่นนี้เป็นการตรวจเชิงลึกทั้งบั๊ก กลไกเหตุการณ์ ความสอดคล้องของค่าจริง และข้อความภาษาไทย โดยเน้นปัญหาที่ทำให้ผู้เล่นได้รับผลผิดจากสิ่งที่หน้าจอบอก เช่น ใช้แรงงานเกินทั้งที่ไม่ได้จัดคนเกิน, ได้รางวัลจากเหตุการณ์ทั้งที่ทรัพยากรไม่พอ, ประชากรเพิ่มซ้ำจากการเกิด และสมุนไพรถูกหักโดยไม่มีผู้ป่วย

## 1. บั๊กสำคัญที่แก้แล้ว

### 1.1 การเกิดเพิ่มประชากรซ้ำ

เดิมเหตุการณ์เกิดมีทั้งผล `population +1` และฟังก์ชันเพิ่มเด็ก ทำให้บางเส้นทางเพิ่ม 2 คน แก้ให้การเกิดเรียกฟังก์ชันเพิ่มเด็กเพียงจุดเดียว จึงเพิ่มจริง 1 คนและสร้างข้อมูลบุคคลครบถ้วน

### 1.2 ตัวเลขใช้แรงงานเกินผิดพลาด

เดิมระบบความเสี่ยงนำ “ค่าผลผลิตรวม” ไปเทียบกับ “จำนวนแรงงานที่มี” ซึ่งเป็นคนละหน่วย จึงอาจแจ้งทำงานเกินและเพิ่มอุบัติเหตุผิดพลาด แก้ให้ใช้ภาระการมอบหมายงานจริงจากแต่ละคนเทียบกับกำลังแรงงานจริง

### 1.3 Autosave บันทึกสถานะไม่ครบ

เดิมการบันทึกอัตโนมัติอาจไม่ทันค่าที่เปลี่ยนในบางส่วน เช่น งานรายคน ตัวเลือกผู้นำ โครงการ หรือนโยบาย และมีโครงสร้างที่เสี่ยงวนอัปเดต แก้ให้บันทึก GameState ทั้งก้อนเมื่อสถานะเปลี่ยน โดยไม่เรียก `setGame` ภายในผลการบันทึก

### 1.4 ใช้สมุนไพรโดยไม่มีผู้ป่วย

คำสั่งต้มสมุนไพรเดิมอาจหักของตามค่าคงที่แม้ไม่มีผู้ป่วย แก้ให้คำนวณจำนวนผู้ป่วยจริง ใช้สูงสุด 2 โดส และไม่ใช้เลยเมื่อไม่มีผู้ป่วย พร้อมรายงานจำนวนคนที่ยังไม่ได้รับการรักษา

### 1.5 สาเหตุป่วย/บาดเจ็บค้างหลังหาย

เมื่อรักษาหายแล้ว ข้อความสาเหตุเก่าอาจยังทำให้ระบบตีความว่าบุคคลนั้นป่วยหรือบาดเจ็บอยู่ แก้การล้างสถานะและสาเหตุให้ตรงกับสุขภาพปัจจุบัน

### 1.6 รับผลดีจากเหตุการณ์ฟรีเมื่อของไม่พอ

เดิมผู้เล่นสามารถเลือกตัวเลือกที่มีค่าใช้จ่ายแม้ทรัพยากรไม่ครบ แล้วบางผลบวกยังเกิดเต็มจำนวน แก้ระบบชำระดังนี้:

- จ่ายครบ: รับผลเต็ม
- จ่ายบางส่วน: ผลบวกที่แบ่งได้ลดตามสัดส่วน
- ผลที่แบ่งไม่ได้ เช่น คน สัตว์ ธง เหตุการณ์ต่อเนื่อง ความทรงจำ หรือคุณลักษณะ จะไม่เกิดจนกว่าจะจ่ายครบ
- ผลเสียยังเกิดตามจริง ไม่ถูกลดเพื่อเอื้อผู้เล่น
- หน้าตัวเลือกแสดงของที่ขาดและเปอร์เซ็นต์ผลประโยชน์ที่จะได้รับ

### 1.7 ID ตัวเลือกเหตุการณ์ซ้ำ

เปลี่ยน ID ของตัวเลือกที่ใช้ชื่อซ้ำในคนละเหตุการณ์ให้เป็นชื่อเฉพาะทั้งหมด ป้องกันปัญหา log, analytics, save migration หรือระบบเชื่อมเหตุการณ์อ้างถึงตัวเลือกผิดรายการ

### 1.8 Package lock อ้าง Registry ภายใน

ไฟล์ล็อกเดิมมี URL สำหรับระบบติดตั้งภายในซึ่งเครื่องผู้ใช้อาจเข้าถึงไม่ได้ แก้ `package-lock.json` ให้ชี้ไปยัง public npm registry และทดสอบ `npm ci` จากไฟล์ล็อกใหม่สำเร็จ

### 1.9 ผู้มาตั้งถิ่นฐานรายปีเพิ่มมากเกินความจุ

เดิมระบบสุ่มเพิ่ม 1–10 คนทุกปีโดยแทบไม่สนที่พักและอาหาร ทำให้ชุมชนล่มจากการเพิ่มคนอัตโนมัติ แก้ให้พิจารณา:

- ระยะพัฒนาของชุมชน
- ที่พักว่าง
- อาหารสำรอง
- ความน่าอยู่/ความไว้วางใจ
- อาจไม่มีคนย้ายเข้าหากชุมชนยังรองรับไม่ได้

## 2. ระบบพักฟื้น

- พักกลางแจ้งลดความล้า 3 หน่วย/เดือน
- มีกองไฟอย่างน้อย 1 กอง ลดความล้าขั้นต่ำ 8 หน่วย/เดือน
- ที่พัก ช่องระบายควัน สุขอนามัย และผลจากผู้นำเพิ่มค่าฟื้นจริง
- ความแออัดลดเฉพาะประโยชน์จากที่พัก ไม่ทำให้ค่าพักต่ำกว่าพักกลางแจ้งหรือกองไฟ
- หน้ารายชื่อแสดงภาพรวมของหมู่บ้านว่าคนที่พักจะลดความล้าเท่าใดในสภาพปัจจุบัน

## 3. Event audit

ตรวจเหตุการณ์ทั้งแบบเขียนตรง แบบสร้างจาก template และเรื่องต่อเนื่อง

| รายการตรวจ | จำนวน | ผล |
|---|---:|---|
| เหตุการณ์ | 336 | ID ไม่ซ้ำ |
| ตัวเลือก | 1,011 | ID ไม่ซ้ำ |
| การอ้างอิงเหตุการณ์ต่อเนื่อง | 127 | ปลายทางครบ |
| ประเมินเงื่อนไข/น้ำหนัก | 1,008 | ไม่เกิดข้อผิดพลาด |
| ทดลองใช้ตัวเลือก | 1,011 | ไม่เกิด exception/NaN |

ตรวจเพิ่มเติมว่า event ที่ให้คน สัตว์ ทรัพยากร สถานะ และเรื่องต่อเนื่องเปลี่ยนค่า GameState จริง ไม่ได้เป็นเพียงข้อความประกอบ

## 4. การปรับข้อความภาษาไทย

ปรับคำให้สอดคล้องกันทั้งหน้าเกมและรายงาน เช่น:

- `Game Over` → `ชุมชนล่มสลาย` / `จบเกม`
- `Outpost` → `ฐานที่มั่นรอง`
- `Gold` → `ทอง`
- `House` → `ตระกูล`
- `Settler` → `ผู้ตั้งถิ่นฐาน`
- `Milestone` → `เหตุการณ์สำคัญ`
- `Grief` → `ความโศกเศร้า`
- `buff/บัฟ` → `ผลเสริม`

ปรับประโยคเหตุการณ์ที่สร้างอัตโนมัติให้มีประธาน กริยา ช่องว่าง และบริบทสัตว์/ครอบครัว/การสำรวจที่อ่านเป็นธรรมชาติขึ้น รวมถึงเปลี่ยนชื่อ NPC ที่ถูกฝังตายตัวในคำบรรยายให้เป็นบทบาทตามเหตุการณ์ เพื่อไม่ให้เรื่องซ้ำหรือขัดกับรายชื่อคนจริง

เหลือคำว่า `JSON` ในข้อความนำเข้า/ส่งออกเท่านั้น เพราะเป็นชื่อรูปแบบไฟล์ที่จำเป็นต่อผู้ใช้

## 5. รายงานจบเดือนและผลผู้นำ

- ผลจากตัวเลือกเหตุการณ์ถูกวัดจากค่าก่อน/หลังจริงและบันทึกในรายงาน
- ผลผู้นำแยกบรรทัดชัดเจน พร้อมผลต่ออาหาร โครงการ วิจัย สำรวจ สุขภาพ หรือความเสี่ยง
- ผลที่แสดงบนหน้าจอเชื่อมกับสูตรคำนวณจริง ไม่ใช่สถานะเพื่อแสดงผลอย่างเดียว
- รายงานเมื่อของไม่พอ การรักษาไม่ครบ หรือผู้มาตั้งถิ่นฐานไม่เข้าเพราะชุมชนรองรับไม่ได้

## 7. ข้อจำกัดและสิ่งที่ควรรู้

- ข้อมูลใน `data/game/*.json` เป็นข้อมูลอ้างอิง/เตรียมพอร์ต ส่วน engine เว็บยังรวมอยู่ใน `app/game/page.tsx`
- การจำลองอัตโนมัติไม่ได้ตัดสินใจสร้างอาคารและวิจัยเหมือนผู้เล่น จึงไม่ใช้ตัดสินความยากสมดุลขั้นสุดท้าย
- Chromium ในสภาพแวดล้อมตรวจสอบถูกบล็อกการนำทาง จึงตรวจภาพด้วย source/CSS และ production response แทนการรับรองจาก screenshot อัตโนมัติ
